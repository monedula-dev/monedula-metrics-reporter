# Releasing

This document is for project maintainers. It describes how to cut a release of `monedula-metrics-reporter`.

## What a release looks like

A release is a Git tag `v<MAJOR>.<MINOR>.<PATCH>` on `main` (e.g., `v0.2.0`). Pushing that tag triggers [`.github/workflows/release.yml`](.github/workflows/release.yml), which:

1. Builds the shadow JAR with the version derived from the tag (`v0.2.0` → `0.2.0`).
2. Runs unit + integration tests.
3. Runs the end-to-end test.
4. Publishes the shadow + sources + javadoc jars to GitHub Packages.
5. Creates a GitHub Release with auto-generated notes and attaches the three jars.

The release is **not** published to Maven Central — see [`build.gradle`](build.gradle)'s commented-out OSSRH block for the wiring that would enable it once the namespace is verified.

## Pre-release checklist

Before tagging, verify all of the following:

- [ ] `main` is green on [CI](https://github.com/monedula-dev/monedula-metrics-reporter/actions/workflows/ci.yml).
- [ ] The latest nightly [Benchmarks](https://github.com/monedula-dev/monedula-metrics-reporter/actions/workflows/benchmarks.yml) run passed and the numbers look sane (JMH callback p50 sub-microsecond, soak heap growth < 100 MB, timeout test green).
- [ ] [`CHANGELOG.md`](CHANGELOG.md) has the new version's entries under `## [Unreleased]`. Each user-visible change since the previous release is captured.
- [ ] No open `Security Advisory` for the previous release.
- [ ] `gradle.properties` does **not** carry a stale `releaseVersion=` override (the release workflow injects the version from the tag).

## Cutting the release

1. **Move `## [Unreleased]` to a versioned section in `CHANGELOG.md`.** Add a new empty `## [Unreleased]` heading above it. Commit:

   ```
   git commit -am "docs: changelog for v0.2.0"
   ```

2. **Tag the release commit and push the tag.** No leading `=`, no `v` typo — the workflow strips the `v` prefix when computing the artifact version.

   ```
   git tag -a v0.2.0 -m "Release v0.2.0"
   git push origin v0.2.0
   ```

3. **Watch the release workflow** at <https://github.com/monedula-dev/monedula-metrics-reporter/actions/workflows/release.yml>. It typically takes ~10 minutes (tests + e2e + publish).

## Post-release verification

After the workflow completes:

- [ ] The GitHub Release at <https://github.com/monedula-dev/monedula-metrics-reporter/releases> has the four primary artifacts plus their Sigstore signatures and certificates:
  - `monedula-metrics-reporter-<version>.jar` + `.sig` + `.pem`
  - `monedula-metrics-reporter-<version>-sources.jar` + `.sig` + `.pem`
  - `monedula-metrics-reporter-<version>-javadoc.jar` + `.sig` + `.pem`
  - `bom.json` (CycloneDX SBOM) + `.sig` + `.pem`
- [ ] The release notes generated by `generate_release_notes: true` reasonably summarise the changes. Edit by hand if they don't.
- [ ] GitHub Packages shows the new version at <https://github.com/orgs/monedula-dev/packages?repo_name=monedula-metrics-reporter>.
- [ ] (Optional) Re-pull the shadow JAR from the Release into a clean directory and verify its SHA-256 matches the one a local `./gradlew shadowJar -PreleaseVersion=<version>` produces. The shadow JAR is configured for reproducible builds, so the hashes should match byte-for-byte.

## Verifying a release artifact (for consumers)

Each artifact ships with a `.sig` and a `.pem` file produced by [Sigstore](https://www.sigstore.dev/) keyless signing during the release workflow. Consumers can verify provenance with `cosign`:

```
cosign verify-blob \
  --signature monedula-metrics-reporter-0.2.0.jar.sig \
  --certificate monedula-metrics-reporter-0.2.0.jar.pem \
  --certificate-identity-regexp 'https://github.com/monedula-dev/monedula-metrics-reporter/.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  monedula-metrics-reporter-0.2.0.jar
```

A successful verification proves the jar was built by this repository's release workflow and has not been tampered since. The same flow works for the sources jar, javadoc jar, and the `bom.json` SBOM.

## If something goes wrong

- **Release workflow fails after publishing the GH Release but before/during package publish.** Re-running the workflow is idempotent for the GH Release (it overwrites) but **not** for GitHub Packages — the version is immutable once published. Delete the tag locally + remotely, increment the patch, retag.
- **Wrong version published.** GitHub Packages versions cannot be deleted via the public UI for organization-owned packages without an admin action. Avoid this by always checking the tag string before pushing.
- **Wrong tag pushed.** If the workflow hasn't started yet, delete the tag immediately:

  ```
  git push origin :refs/tags/v0.2.0
  git tag -d v0.2.0
  ```

  If the workflow has already started, let it finish and treat the result as final, or accept a 0.2.1 patch release.

## Versioning policy

This project follows [Semantic Versioning](https://semver.org/spec/v2.0.0.html). The public API for purposes of SemVer is everything under `dev.monedula.metricsreporter` that an integrator could see (the `MetricsReporter` SPI implementation, the configuration keys exposed via `OtlpMetricReporterConfig`, and the published Maven coordinates). The shaded third-party packages under `dev.monedula.metricsreporter.shaded.*` are implementation detail and may change without a major bump.

## Repository setup (one-time)

For discoverability, the following GitHub topics should be set on the repo via **Settings → About → Topics**:

- `kafka`
- `apache-kafka`
- `opentelemetry`
- `otlp`
- `observability`
- `metrics`
- `prometheus`
- `java`
- `metrics-reporter`

These cannot be committed via Git; they're a one-time configuration in the repository UI.

Also verify in the same UI:

- **Issues:** enabled.
- **Security advisories:** enabled (allows the private vulnerability reporting flow in [SECURITY.md](SECURITY.md)).
- **Branch protection on `main`:** require the `CI` and `Validate Gradle wrapper` checks before merge; require pull request reviews.
- **Default branch:** `main`.
